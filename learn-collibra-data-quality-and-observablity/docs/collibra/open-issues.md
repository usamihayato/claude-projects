## 1. ExpressRoute 共有帯域の逼迫リスク

| 項目 | 内容 |
|---|---|
| **ステータス** | 🔴 未着手 |
| **優先度** | 高 |
| **影響範囲** | Standalone / AKS 共通（構成手法に依存しない社内インフラ制約） |
| **発生条件** | スキャン対象データソースにオンプレ DB が含まれる場合 |

### 背景・課題内容

ExpressRoute は全社共通リソースであり、DQ スキャン時に大量データ（最大 2TB / 回）がオンプレ → Azure 間を流れることで、他システムの通信帯域を圧迫するリスクがある。

```
オンプレ DB ──ExpressRoute（共有）──▶ Azure DQ Agent（VM / AKS）
                 ↑
          全社共通リソース。他システムと帯域を共有
```

### 制御手段の整理

#### A. Collibra DQ アプリケーションレベル

| 手段 | 効果 | 副作用 | 実現難易度 |
|---|---|---|---|
| **スキャン行数上限（`limit` 設定）** | 読み込みデータ量を削減 | 品質チェックの網羅性が下がる | ★☆☆ 低 |
| **サンプリング（Sampling）モード** | 全件の数〜数十% の読み込みで済む | 異常検知精度が低下する可能性あり | ★☆☆ 低 |
| **スキャン時間帯のスケジュール制御** | 深夜・休日など混雑しない時間帯に集中 | スキャン結果の鮮度が遅れる | ★☆☆ 低 |

#### B. Spark レベル

| 手段 | 効果 | 副作用 | 実現難易度 |
|---|---|---|---|
| **JDBC 並列接続数の制限**（`sparkexecutorinstances=1`） | 同時 JDBC 接続数を抑えてピーク帯域を平滑化 | スキャン完了時間が延びる | ★★☆ 中 |
| **Executor コア数の制限**（`sparkexecutorcores=2`） | 同上 | 同上 | ★★☆ 中 |

#### C. OS レベル（Linux `tc`）

| 手段 | 効果 | 副作用 | 実現難易度 |
|---|---|---|---|
| **`tc`（traffic control）による帯域上限設定** | bps 単位で正確に制限可能（唯一の厳密制御手段） | VM 全体のアウトバウンドに影響。設定ミスで通信断のリスク | ★★★ 高 |

```bash
# 設定例: オンプレ宛トラフィックを 200Mbps に制限
tc qdisc add dev eth0 root handle 1: htb default 10
tc class add dev eth0 parent 1: classid 1:1 htb rate 200mbit
tc filter add dev eth0 protocol ip parent 1:0 prio 1 \
  u32 match ip dst <オンプレIPレンジ>/8 flowid 1:1
```

> **AKS の場合**: Node レベルでの設定が必要。DaemonSet として管理する方式が現実的。

#### D. Azure / ネットワークレベル

| 手段 | 効果 | 副作用・制限 | 実現難易度 |
|---|---|---|---|
| **ExpressRoute QoS（DSCP マーキング）** | 業務系トラフィックを優先。混雑時に DQ スキャンが押しのけられる | 帯域を「制限」するのではなく「優先度を下げる」制御。オンプレ側ルーター設定も必要（**ネットワーク部門との協調必須**） | ★★★ 高 |
| **Azure Monitor + Automation Runbook による自動停止** | ExpressRoute 使用率が閾値超過時に DQ VM を自動停止 | リアクティブな対応（事前抑制ではない） | ★★☆ 中 |
| **NSG による帯域制限** | **不可** | NSG は許可/拒否のみ。bps 制限機能なし | — |
| **Azure Firewall による帯域制限** | **不可** | L7 フィルタリングのみ。スロットリング機能なし | — |
| **ExpressRoute 回線帯域の縮小** | **不可** | 帯域変更は増速のみ可能。縮小不可 | — |

### 推奨対応方針（案）

```
フェーズ1（構築時・必須）:
  → サンプリング設定 or 行数上限を DQ ジョブにデフォルト設定
  → スキャンスケジュールを深夜・休日に限定

フェーズ2（運用開始後・推奨）:
  → Azure Monitor で ExpressRoute 帯域使用率を継続監視
  → 閾値アラート + Automation Runbook による自動停止を実装

フェーズ3（必要に応じて）:
  → Linux tc による OS レベルの帯域上限設定
  → ExpressRoute QoS 設定（ネットワーク部門との協議）
```

### 残課題・確認事項

- [ ] ExpressRoute 回線の現在の契約帯域幅と平均使用率の確認（ネットワーク部門）
- [ ] サンプリングモードの適用可否を Collibra 担当者に確認（品質精度への影響）
- [ ] 深夜スキャンの許容性確認（業務要件として翌朝までに結果が必要か）
- [ ] Linux `tc` 設定の運用管理体制（Standalone の場合）/ DaemonSet 管理（AKS の場合）
- [ ] ExpressRoute QoS 設定の実施可否（ネットワーク部門・オンプレ側ルーター担当との合意）
