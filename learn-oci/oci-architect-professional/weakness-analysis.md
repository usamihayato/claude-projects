#### 1. 高可用性とディザスタリカバリ（約20問誤答）

- Test1: Q10, Q19, Q20, Q22, Q35, Q46, Q48
- Test2: Q6, Q26, Q32
- Test3: Q1, Q2, Q33, Q37, Q42
- Test4: Q16, Q21, Q33
- Test5: Q25, Q44, Q48

| パターン | 具体的な誤り |
| --- | --- |
| FD（フォールト・ドメイン）の役割を軽視 | AD間分散だけでは不十分。各AD内でFDにも分散が必要（T4-Q16） |
| RACをリージョン間で使おうとする | RACは単一リージョン内専用。リージョン間はData Guard/GoldenGate（T4-Q21、T5-Q44誤選択） |
| Block VolumeはAD固有と覚えていない | 異なるADのインスタンスで同一Block Volumeを共有できない（T1-Q22） |
| Object StorageのCRRは「自動」と思い込んでいる | リージョン間複製は手動でレプリケーション・ポリシーを設定する必要がある（T1-Q35、T5-Q48） |
| KMS暗号化キーのクロスリージョン問題を見落とす | キーはリージョン固有。DR先でも復号できるようキー管理が必要（T1-Q48） |
| データ主権要件でクロスリージョン複製を選んでしまう | 国外持ち出し禁止の場合、リージョン内のADG/バックアップで対応（T2-Q26、T5-Q21） |
| Active Data Guard と Data Guard の違いが曖昧 | Active Data Guard = スタンバイからの読み取り可能（低レイテンシ読取） + 自動フェイルオーバー（T3-Q33） |

```text
リージョン内HA    → AD分散 + FD分散 + ロードバランサ
リージョン障害DR  → マルチリージョン + Data Guard / Object Storage CRR（手動設定）
低レイテンシ読取+自動DR → Active Data Guard
コスト重視の緩やかなRPO → スクリプト+Object Storageでの定期転送
データ主権あり   → クロスリージョン禁止、リージョン内Autonomous Data Guard
```

#### 2. OCIデータベース（約21問誤答）

- Test1: Q6, Q30
- Test2: Q6, Q11, Q34, Q36, Q50
- Test3: Q1, Q2, Q35, Q50
- Test4: Q21, Q46
- Test5: Q1, Q4, Q28, Q44
- Oracle University: Q7, Q10, Q11, Q12

| パターン | 具体的な誤り |
| --- | --- |
| ATPとADWの使い分けが曖昧 | ATP=OLTP+混合ワークロード。ADW=分析特化。画像・動画はObject Storageへ（T1-Q6、T1-Q30、T2-Q34） |
| Autonomous Databaseの自動バックアップを知らない | ADBは標準で60日自動バックアップあり。手動不要（T2-Q6） |
| ADB接続は固定IPではなく接続文字列（Wallet）を使う | 固定IP依存はフェイルオーバー時に接続失敗する（T2-Q6） |
| RACは「強一貫性+単一リージョン内」と覚えていない | 金融アプリの強一貫性 = RAC（同一AD内共有ストレージ）（T3-Q35） |
| MySQL DBSにData Guardは使えない | MySQL DR = リード・レプリカ or クロスリージョン・バックアップ（T4-Q46） |
| Exadata = オンプレRAC移行の最適解と気づかない | RACそのままクラウド化 = Exadata DB System（T2-Q36） |
| Database Management Serviceの役割を知らない | 異種混合DBフリートの統合監視・管理（T2-Q11、T5-Q28） |
| ADWの組み込みMLを見落とす | ADW = 分析+Oracle Machine Learning (OML)内蔵（T5-Q4） |
| Autonomous DBのオートスケーリングを選ばない | 動的ワークロード変動 = Autonomous DB + Auto-scaling（T3-Q50、T5-Q50） |
| ATP-S / AJD / ATP-Dの使い分けが曖昧 | MongoDB代替=AJD、Oracle E-Business Suite=ATP-D専用（OU-Q10） |
| ATP接続方法を知らない | ATP接続はWallet/ORDSが正解。CreateConnection APIはコンテナ直接接続に不適（OU-Q11） |
| 手動スケールアップの問題点を見落とす | 手動でのCPUスケールアップはプロビジョニングに時間がかかりピーク時に遅延（OU-Q12） |
| DBバックアップの前提条件を知らない | Object StorageバックアップはHTTPS（TCP443）使用、DBはARCHIVELOGモード必須（OU-Q7） |

```text
ATP   = OLTP / 混合ワークロード / 自動バックアップ / 自動スケーリング（基本OCPU数の最大3倍）
ADW   = 分析 / ML(OML)内蔵 / リアルタイムレポート
AJD   = Autonomous JSON Database = MongoDBの代替
ATP-D = 専用インフラ = Oracle E-Business Suite専用
RAC   = 強一貫性 / 単一リージョン内のみ / 共有ストレージ
Exadata  = オンプレRACの最適移行先
MySQL DR ≠ Data Guard（Oracle DBのみ）
ADB接続  = Wallet ＋ 接続文字列 または ORDS（CreateConnection APIは不可）
DBバックアップ = HTTPS(TCP443) ＋ ARCHIVELOGモード必須
Database Management = 異種混在DBの統合管理ツール
```

#### 3. OCIへのワークロードの移行（約14問誤答）

- Test1: Q28, Q31, Q34, Q49
- Test2: Q36, Q45
- Test3: Q31, Q39, Q41, Q45
- Test4: Q19
- Test5: Q12, Q24, Q37

| パターン | 具体的な誤り |
| --- | --- |
| 最小ダウンタイム移行 = DMSと結びつかない | OCI DMS = ZDM技術+GoldenGate内包でリアルタイム同期（T1-Q31、T5-Q12） |
| Data Pump = 大容量移行に使えないと分かっていない | Data PumpはOracle専用かつ大容量では時間がかかりすぎる（T1-Q31、T1-Q28） |
| NoSQL移行 = 手動スクリプトが現実的と気づかない | DMS/Data PumpはRDB専用。NoSQLは個別対応（T1-Q28） |
| 大規模移行のボトルネック = ネットワーク帯域 | VMフォーマット変換やストレージ互換はツールが自動対応（T1-Q34） |
| 50TB移行 = Data Transfer Service（ディスク配送） | ネットワーク経由では48時間に収まらない（T3-Q39） |
| Archive Tier = RTO要件違反と気づかない | Archive取り出しには数時間かかる。RTOが短いならStandard必須（T3-Q39） |
| 移行情報収集で「規制・コンプライアンス」を後回しにする | 金融/医療移行では規制（PCI-DSS等）とライセンスが設計の根幹（T3-Q31、T5-Q24） |
| ツールバージョンとリポジトリ構成を軽視する | 開発環境移行ではCI/CDツールのバージョン互換が最優先（T3-Q45、T5-Q37） |
| 複雑な依存関係 = ハイブリッド段階移行が最適と気づかない | 一気移行はリスク大。K8sマネージドサービス活用の段階移行が正解（T3-Q41） |

```text
最小ダウンタイム        → OCI DMS（ZDM+GoldenGate）
大容量（数十TB）移行    → Oracle Data Transfer Service（ディスク配送）
迅速取得が必要          → Object Storage Standard（Archiveは絶対NG）
移行情報の優先順位      → 規制/コンプライアンス > ライセンス > 技術スタック
複雑なアプリ            → ハイブリッドアプローチ + マネージドサービスへ段階移行
NoSQL移行               → カスタムスクリプト（DMSはRDB専用）
```

#### 5. セキュリティ（9問誤答）

- Test1: Q48
- Test2: Q5
- Test3: Q27
- Test5: Q7, Q21, Q40, Q43
- Oracle University: Q3, Q14

| パターン | 具体的な誤り |
| --- | --- |
| KMSキーのリージョン固有性を忘れる | CRR環境でキー管理を怠るとデータにアクセス不能（T1-Q48） |
| Explicit Deny の優先度を知らない | OCI認可: Deny > Allow（どんなAllowより優先される）（T3-Q27） |
| バケット・ポリシーとIAMポリシーの役割分担 | IAM=誰が、バケットポリシー=どのサービス/条件でを細かく制御（T5-Q43） |
| Cloud Guard + Data Safe の組み合わせを覚えていない | GDPR+高度なDBセキュリティ → Cloud Guard + Data Safe（T2-Q5） |
| パブリックバケット ≠ 権限エラーの原因 | パブリックなら認証不要のはず → 権限エラーの原因にならない（T5-Q40） |
| Block Volume顧客管理キーの手順を知らない | ①Vault作成 → ②マスター暗号化キー作成 → ③Block Volumeへ割り当て（OU-Q3） |
| KMSの2エンドポイントを混同する | 管理エンドポイント（キー管理）と暗号エンドポイント（暗号化操作）は別。URLの`management`/`crypto`で判別（OU-Q14） |

```text
KMS管理エンドポイント  → キー自体の作成・ローテーション・削除
KMS暗号エンドポイント → 実際の暗号化・復号操作（URLに"crypto"が含まれる）
Block Volume暗号化     = Vault作成 → キー作成 → ボリュームへ割り当て（3ステップ）
Explicit Deny          > どんなAllow（最優先で拒否）
```

#### 6. 監視ソリューション（2問誤答）

> ⚠️ Udemyでは誤答ゼロだったが、公式試験で発覚。「得意分野」ではなかった。

| パターン | 具体的な誤り |
| --- | --- |
| エージェント構成の役割を知らない | OCIロギングのエージェント構成でインスタンスからのログ収集ルールを定義。サービス・コネクタ・ハブでObject Storageへ自動転送（OU-Q2） |
| OCIイベントでできないことを誤る | OCIイベントはリソース状態変化のトリガー用。モニタリングアラームの取得やオートスケーリング呼び出しはモニタリング・サービスの機能（OU-Q5） |

```text
エージェント構成        → ログ収集ルール定義（何を集めるか）
サービス・コネクタ・ハブ → ログの自動ルーティング先（Object Storage / Streaming 等）
OCIイベント            → リソース状態変化に基づくアクション（≠ モニタリングアラーム）
```

#### 7. ストレージ詳細（3問誤答）

| パターン | 具体的な誤り |
| --- | --- |
| File Storageのビッグデータ適合性を知らない | OCIファイル・ストレージは動的拡張で高スループット・低レイテンシ。ビッグデータアプリに最適（OU-Q4） |

```text
Functions課金      = 呼出し回数 + 実行時間 + メモリ量（3要素）
```

## 共通の失点パターン（全カテゴリ横断・更新版）

1. **「単一リージョン内HA」と「リージョン障害DR」を混同する** → HAはリージョン内、DRはリージョン間
2. **RACをリージョン間で使おうとする** → RACは物理的に近いノード間のみ
3. **「自動」と「手動設定が必要なもの」を間違える** → CRRは手動、ADBバックアップは自動
4. **コスト効率のトレードオフを誤る** → RPO/RTO要件に合ったコスト最小解を選ぶ
5. **移行の必須情報収集で技術詳細より規制を後回しにする** → 規制・コンプライアンスが最優先
6. **サービスの役割境界を混同する** → イベント≠モニタリング、管理エンドポイント≠暗号エンドポイント
7. **同一サービス内の派生型を知らない** → Autonomous DBはATP-S / AJD / ATP-Dで用途が異なる
