
Oracle Functionsは、サーバーのプロビジョニングや管理を行わずにコードを実行できるサーバーレス・プラットフォームです。使用したリソースについてのみ支払います。これは、「呼出し回数」や「実行時間(秒)×メモリ量(GB)」で算出され請求されます。
つまり、ファンクションの使用料の算出に使用される情報はファンクションの呼出し回数、実行時間、使用されるメモリ(RAM)の量の3つの要素ですが、メモリ(RAM)の量は、ファンクションのモニタリング・メトリックではないため、ここでは選択肢として適切ではありません。

エージェント構成は、OCIロギングの機能で、インスタンスから収集するログを定義することができます。これにより、Infoレベルのログが確実に取得され、処理と分析のために OCI ロギング・サービスに転送されるようにできます。OCIサービス・コネクタ・ハブは、長期保存のためにOCIロギングからOCIオブジェクト・ストレージへのログの自動ルーティングを可能にします。サービス・コネクタを作成することで、手動で操作することなくログを効率的にアーカイブし、コンプライアンスと監査要件を満たすことができます。


顧客管理キーを使用してOracle Cloud Infrastructure (OCI)のブロック・ボリュームを暗号化するには、次のタスクが必要です:
1. ボールトの作成: 顧客の暗号化キーを安全に管理するために、OCIでKey Management Vaultを設定します。
2. マスター暗号化キーの作成: ボールト内に顧客管理キーを生成し、そのプロパティおよび暗号化設定を定義します。
3. マスター暗号化キーの割当て: 顧客管理キーをブロック・ボリュームに関連付け、ボリュームがこのキーを使用して暗号化されていることを確認します。
ボールトを作成し、マスター暗号化キーを生成し、それをブロック・ボリュームに割り当てることで、保険会社は規制要件に準拠し、重要な財務データを保護するために使用される暗号化キーを完全に制御できます。


OCIファイル・ストレージ・サービスは動的に拡張して高いスループットをサポートするため、ビッグデータ・アプリケーションに最適です。分析ワークロードに必要な低レイテンシのファイル操作を提供します。

OCIイベントではイベントに基づいてアクションをトリガーできますが、モニタリング・アラームの取得およびコンピュート・インスタンスの自動スケーリングの呼出しは、OCIモニタリング・サービスの機能とより密接に関連しており、イベント・サービスのユース・ケースではありません。

アップロードされるオブジェクトの最大サイズは10TiBです。各パーツに使用するパーツ番号を決めます。パーツ番号の範囲は1～10,000です。連続した番号を割り当てる必要はありませんが、オブジェクト・ストレージはパーツ番号を昇順に並べてオブジェクトを構成します。オブジェクト・パーツの作成が完了したら、CreateMultipartUpload REST APIコールを実行して、マルチパート・アップロードを開始します。オブジェクト名とオブジェクトのメタデータを指定します。オブジェクト・ストレージは一意のアップロードIDで応答します。このアップロードIDは、このマルチパート・アップロードに関連するすべてのリクエストに含める必要があります。オブジェクト・ストレージはまた、アップロードをアクティブとしてマークします。アップロードは、明示的にコミットするか中止するまでアクティブなままです。



バックアップが実行されているデータベース・ホストとOCIオブジェクト・ストレージ間の接続が失敗すると、オブジェクト・ストレージへのバックアップが失敗します。オブジェクト・ストレージへの接続はHTTPS(TCPポート443)を使用するので、デフォルトのセキュリティ・リストを変更する際はエグレス・ルールの記述に注意が必要です。dcsagentの再起動は必要に応じて行いますし、バックアップ時のデータベースは実行中である必要があります。また、データベースのアーカイブモードはARCHIVELOGに設定されている必要があります。

リスナー2には、バックエンド・セットBの正確なルールがあります。また、パス・ルート・セットには、URI /tameを含む任意の文字列をバックエンド・セットBにルーティングする必要があることを示す一致ルールがあります。


特定のコンパートメントに対して予算を作成し、そのコンパートメント内のすべてのリソースの全体的な支出を追跡することができます。また、共通のタグを共有するリソースの予算を作成することもできます。予算範囲は、「コンパートメント」または「タグ」のいずれかにすることができます。以前は「タグ」の使用は「コスト・トラッキング・タグ」に限定されていましたが、2024年6月以降はすべての定義済みタグが使用できるようになっています。


ATP-Sは、OCIのフルマネージド・データベース・サービスで、コンピュート構成の3倍まで拡張可能な自動スケーリング機能を備え、ピーク時の要件に自動的に対応します。予想されるワークロードを予測することなくOCIにアプリケーションを展開する新興企業に最適です。さらに、データベースが使用されているときは稼働させ、使用されていないときはシャットダウンしてコストを節約することができます。MongoDBの代替であればATP-SではなくAutonomous JSON Database(AJD)が最適です。また、Oracle E-Business SuiteはATP-D(専用インフラ)にのみ対応しているのでこの場合は適した選択肢ではありません。



Kubernetes シークレットを使用して、ATPインスタンスのOCIDとOCI API資格情報で環境変数を構成し、CreateConnection API エンドポイントを使用して接続を試みる方法は有効ではありません。CreateConnection API は、コンテナ化されたアプリケーションからATPに実行時に直接接続するようには設計されていません。代わりに、ATP接続では通常、ATPインスタンスのウォレット(Wallet)または Oracle REST Data Services を使用する必要があるため、他のオプションが有効です。

ピーク時にCPUコア数およびストレージにスケール・アップすることは、追加のリソースのプロビジョニングに時間がかかるため、ピーク時のアプリケーション・レスポンス時間が長くなる可能性があり、実用的ではありません。

ピーク時間に必要な最大CPU容量を特定し、ATP-SデータベースのCPUコア数をその数に設定することは、オフピーク時には必要以上に多くのCPUコアを支払う必要があるため、コストが高くなります。自動スケーリングでは指定したCPUコア数を下回るスケーリングは自動的に行われません。

Azure ExpressRouteでOCI FastConnectをセットアップする場合、接続はOCIとAzure間のクロス・クラウド相互接続によって確立されます。このプロセスには、OCI側でプロビジョニングされるOCI FastConnect仮想回線とAzure側でプロビジョニングされるAzure ExpressRoute回線という2つのコンポーネントが含まれます。
これら両方のコンポーネントは相互に依存しています。ExpressRoute回線は、関連する FastConnect 仮想回線が存在する限り、「Provisioned」または「Enabled」状態のままです。OCI FastConnect仮想回線を削除せずにAzure Resource Groupを削除しようとしても、ExpressRoute回線は自動的に削除されません。

OCI KMS(Key Management Service)は、管理エンドポイントと暗号エンドポイントの2つのエンドポイントを提供します。暗号化操作には、暗号エンドポイントを使用する必要があります。エラーが発生したのは、開発者が暗号エンドポイントではなく、管理エンドポイントを使用したためです。管理エンドポイントであることはhttps://bbptfrr5aaeuk-management.kms.us-ashburn-1.oraclecloud.comのmanagement部分から判別できます。暗号エンドポイントの場合、この部分がcryptoになります。
