問題1
不正解
グローバル物流会社が、高度なデータベース機能とクラウドのスケーラビリティを活用するために、在庫管理システムをOCIにアップグレードすることを計画しています。このシステムは同社の業務に不可欠であり、災害発生時のダウンタイムとデータ損失を最小限に抑えるため、目標復旧時点 (RPO) 2時間、目標復旧時間 (RTO) 30分を必要としています。ソリューションは、異なるリージョン間での業務継続性とデータの可用性を確保しなければなりません。コスト効率を考慮しつつ、これらのRPOおよびRTO要件を最も満たす構成はどれですか？

A. プライマリリージョンにベアメタルインスタンスで3ノードのOracle RACを構築し、セカンダリリージョンのVM上の3ノードOracle RACへのデータ複製の目的でOracle GoldenGateを手動構成する。

C. プライマリリージョンに1ノードのVM Oracle Databaseをデプロイし、セカンダリリージョンの1ノードVM Oracle Databaseに対して Oracle Data Guard を使用してリアルタイム・データレプリケーションを行う。

回答は不正解です
D. トランザクション処理のために一リージョンに Autonomous Database (ATP) を実装し、別のリージョンの Autonomous Database へのリアルタイム・データレプリケーションを Oracle GoldenGate で処理する。

正解
B. プライマリリージョンのVM上に2ノードの Oracle RAC を構成し、セカンダリリージョンの2ノード Oracle RAC に対して Oracle Active Data Guard を使用して同期データレプリケーションを行う。

全体的な説明
【詳細解説】

正解（B）の根拠: RTO 30分 という非常に短い要件を満たすには、障害発生時に「DBを立ち上げる」時間すら惜しいため、待機系が常に起動している Active Data Guard が最適です。また、RAC を各リージョンで使用することで、リージョン内での単一ノード障害時もダウンタイムなしで継続でき、全体の可用性が大幅に向上します。VMベースの2ノードRACは、性能とコストのバランスが取れています。

不正解の理由:

A: ベアメタル3ノード構成は非常に高額であり、「コスト効率」の文句に反します。またGoldenGateの手動管理は運用負荷が高くなります。

C: シングルノードでは、リージョン内のAD故障やパッチ適用時にダウンタイムが発生し、業務継続性の要件（ミッションクリティカル）に不安が残ります。

D: Autonomous Databaseを使用する場合、わざわざGoldenGateを別途構成するよりも、標準機能の Autonomous Data Guard を使う方がシンプルかつ低コストです。

【重要ポイント】

厳しいRTO（30分以内）には、迅速なフェイルオーバーが可能な Active Data Guard 構成が推奨される。

問題2
不正解

回答は不正解です
B. 1つのリージョンにプライマリ、別のリージョンにスタンバイを配置する Data Guard を構成する。高可用性のためにオートフェイルオーバーをセットアップする。

【詳細解説】

正解（D）の根拠: * コスト効率: Data Guard (B) や GoldenGate (C) は、ターゲット側で常時起動しているデータベースインスタンス（およびライセンス）のコストがかかります。D は変更分（5% = 100GB/日）のみを安価な Object Storage に保管し、必要な時だけ処理（Functions等）を行うため、待機系の維持コストを最小化できます。

RPO 4時間: 4時間ごとの定期的な差分転送により、RPO要件を充足可能です。

A: 「同じリージョン内」での保管は、リージョン全体の全損からデータを保護できず、災害復旧の目的を果たせません。

B, C: これらは RPO ほぼゼロの高度なソリューションですが、本設問の「4時間」という緩やかな RPO 要件に対しては、維持コストが高すぎる（オーバーキル）と判断されます。

【重要ポイント】

災害復旧の設計では、RPO/RTOの要件強度とコストのバランス（コスト効率）が常に問われる。

問題10
不正解
AIアプリケーションを専門とする革新的なソフトウェア開発会社が、クライアントにサービスとして提供するために、最近 OCI 上に複雑な機械学習モデルをデプロイしました。デプロイメント・アーキテクチャには、モデル実行用の OCI Compute インスタンス、トレーニングデータセットとモデルアーティファクト保存用の OCI Object Storage、モデルへの API リクエストを管理する Load Balancer、およびアプリケーションデータ保存用の OCI Autonomous Database が含まれています。モデルの機能を実演するために、同社は潜在的なクライアント向けにライブデモを開催しました。デモ中、ユーザーがモデルにクエリを実行した際に応答速度が遅くなる事象が発生しました。この問題の理由として考えられる2つの原因はどれですか？

選択は正解です
A. 機械学習モデルを実行している OCI Compute インスタンスが CPU とメモリの上限に達したが、オートスケーリングが構成されていなかったため、負荷を処理するためのリソース追加ができなかった。

B. OCI Object Storage でサービスの中断が発生し、トレーニングデータセットとモデルアーティファクトへのアクセスに遅延が生じた。

選択は不正解です
C. Load Balancer が単一のバックエンドサーバーのみを使用するように構成されていたため、着信 API リクエストを複数のサーバーに分散できなかった。

正しい選択
E. Autonomous Database がワークロードに対して低すぎる CPU とストレージ制限で構成されていたため、データの取得と保存操作においてボトルネックが発生した。

D. ネットワーク ACL（アクセス制御リスト）が誤って構成されており、Compute インスタンスと Autonomous Database 間の通信がブロックされていた。

全体的な説明
【詳細解説】

正解（A, E）の根拠: * A: 機械学習の推論（Inference）は計算リソースを大量に消費します。デモのように同時アクセスが増える場面で、Auto-scaling が未設定だと既存インスタンスのリソース（CPU/RAM）が枯渇し、処理待ちが発生して応答が遅延します。

E: アプリケーションデータを管理する Autonomous Database の OCPU（CPU）が不足していると、クエリ処理や結果の保存が追いつかず、システム全体のレスポンスを悪化させるボトルネックとなります。

不正解の理由:

B: Object Storage は極めて高い耐久性と可用性を持ち、事前の通知なしに「遅延」を招くような中断は考えにくいです。また、通常、推論実行時にはモデルファイルはロード済みであり、ストレージへの継続的なアクセスは発生しません。

C: ロードバランサに単一サーバーしか紐付いていないのは「構成上の不備」ではありますが、OCI のサービスとしての制限ではなく、1台のサーバーで捌ける範囲であれば「遅延」の直接原因とは断定できません（AやEの方が根本的なリソース不足を指しています）。

D: ACL で「ブロック」されている場合、応答が「遅い」のではなく、接続エラー（タイムアウト等）となり通信自体が失敗します。

【重要ポイント】

パフォーマンス遅延の主因は、Compute層のオートスケーリング未設定、またはデータベース層のリソース不足（OCPU不足）であることが多い。

問題27
不正解
あるプロジェクトのために OCI Object Storage を構成している際、セキュリティを強化するためにバケットに厳格なアクセス・ポリシーを適用することにしました。新しいポリシーを適用した後、ユーザーからバケット内のデータにアクセスできなくなったという報告がありました。このアクセス問題の最も可能性の高い理由は何ですか？

回答は不正解です
B. IAM ポリシーが新しいバケット・ポリシーを反映するように更新されておらず、両者の間で競合が発生している。

正解
C. バケットに適用された新しいポリシーが、アカウント管理者以外のすべてのユーザーへのアクセスを明示的に拒否（Deny）している。

A. ユーザーの認証情報が、ポリシーの更新とちょうど同じタイミングで期限切れになった。

D. ポリシーの更新中にバケットが誤って別のコンパートメントに移動され、権限の問題が発生した。

全体的な説明
【詳細解説】

正解（C）の根拠: Object Storage には、IAM ポリシーとは別に、バケットごとに設定できる**「バケット・ポリシー」**があります。ここで条件（Condition）の設定を誤り、「明示的な拒否 (Explicit Deny)」を含めてしまうと、IAM で許可されていてもアクセスが拒否されます（OCI では Deny は Allow よりも常に優先されます）。「厳格なポリシーを適用した直後に全員アクセス不可」という状況に最も合致する原因です。

不正解の理由:

A: タイミングとしてはあり得ますが、全ユーザーが同時に期限切れになるのは極めて不自然です。

B: IAM ポリシーとバケット・ポリシーは「論理和（どちらかで許可されていれば良い）」として動作しますが、どちらかに Deny があれば拒否されます。「反映されていない」ことによる競合が即座にアクセス遮断を招くわけではありません。

D: バケットの移動はポリシー適用とは別の操作であり、問題文の「ポリシーを適用した後」という文脈から外れます。

【重要ポイント】

OCI の認可ロジックでは「明示的な拒否（Explicit Deny）」は、いかなる「許可（Allow）」よりも優先される。

問題35
不正解
OCIアーキテクトとして、金融サービス向けアプリケーションのために、OCI内で高可用かつスケーラブルなデータベース構成を設計しています。アプリケーションは、オートスケーリング、自己修復能力、およびデータベースインスタンス間での強一貫性を求めています。これらの要件を満たしつつ、パフォーマンスと管理性を最適化するために、どのデータベースソリューションの導入を推奨しますか？
