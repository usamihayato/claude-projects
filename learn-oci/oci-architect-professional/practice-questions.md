### Q1
ある企業がOCI上で重要なWebアプリケーションを運用しています。同じリージョン内の2つの可用性ドメイン（AD）にComputeインスタンスを分散し、ロードバランサを構成しました。しかし定期メンテナンス中に断続的なダウンタイムが発生しています。最も可能性の高い原因はどれですか？

A. ロードバランサのヘルスチェック間隔が短すぎる  
B. 各AD内でフォールト・ドメイン（FD）を考慮したインスタンス配置がされていない  
C. ロードバランサが単一ADにしか対応していない  
D. ComputeインスタンスにパブリックイPアドレスが割り当てられていない

<details>
<summary>▶ 解答・解説</summary>

**正解: B**

AD間で分散しても、各AD内のインスタンスが同一FDに集中していると、そのFDのラックや電源系統に障害が発生した際に全インスタンスが影響を受ける。真のHAには「AD間分散」と「AD内FD分散」の両方が必要。

- A: 攻撃的なヘルスチェックは一時的な切り離しを招くが、複数ADがある場合は他ADがカバーするためダウンタイムには直結しない
- C: OCI LBはリージョナルサービスであり複数ADに対応している
- D: パブリックIPはセキュリティ問題だが可用性とは別問題

**重要ポイント**: 高可用性 = AD間分散 ＋ **FD分散** のセット
</details>

### Q2
金融機関がOCI上のOracle Databaseに対してData Guard構成を組んでいます。同一リージョン内のプライマリとスタンバイ間で低レイテンシの読み取りアクセスが必要で、ADが落ちた際の自動フェイルオーバーも求められます。最適な構成はどれですか？

A. Oracle Data Guard（Maximum Protection モード）  
B. Oracle Active Data Guard  
C. Oracle GoldenGate  
D. Oracle RAC（2ノード構成）

<details>
<summary>▶ 解答・解説</summary>

**正解: B**

Active Data Guardは通常のData Guardに加え、スタンバイDBからの**読み取り専用アクセス**が可能。これにより読み取りの負荷分散と低レイテンシを実現しながら、障害時は自動フェイルオーバーで切り替え可能。

- A: Data Guard（Maximum Protection）はデータ損失ゼロだが、スタンバイからの読み取りはできない
- C: GoldenGateはアクティブ・アクティブの双方向レプリケーション用。読取分散だけが目的なら過剰
- D: RACは単一リージョン内・共有ストレージ前提。ADをまたいだスタンバイ読取とは異なる概念

**重要ポイント**: **「スタンバイ読取 + 自動DR」= Active Data Guard**
</details>

### Q3
企業がOCI Object Storageを使ってミッションクリティカルなデータを管理しています。リージョン障害からデータを守るため、別リージョンへの複製を計画しています。正しい説明はどれですか？（2つ選択）

A. Object Storageはデフォルトで自動的に別リージョンへデータを複製する  
B. クロスリージョン・レプリケーション（CRR）はユーザーが手動でポリシーを設定する必要がある  
C. CRRを設定すれば、ソースバケットのオブジェクト削除は自動的に複製先に同期される  
D. Object Storageはリージョン内の複数ADに自動的にデータを分散して保存する  
E. Archive TierのオブジェクトはリアルタイムでCRRが実行される

<details>
<summary>▶ 解答・解説</summary>

**正解: B、D**

- B: CRRは**手動でレプリケーション・ポリシーを設定**する必要がある。デフォルトではリージョン間の複製は行われない
- D: Object Storageはリージョン内では複数ADに自動分散保存する（高耐久性の基盤）

- A: リージョン間の自動複製はデフォルトでは**行われない**
- C: 削除の同期はCRRの設定によるが、一般的にはソースの削除は複製先に同期される。ただしこの選択肢は問題文の文脈では正解の核心ではない
- E: Archive TierのオブジェクトのリアルタイムCRRはサポート対象外の場合がある

**重要ポイント**: Object Storageの**リージョン内冗長は自動**、**リージョン間冗長は手動設定**
</details>

### Q4
ある多国籍企業がOCIで重要なアプリケーションを運用しており、ヨーロッパのデータ主権法（域外持ち出し禁止）に準拠する必要があります。同時に、システムの高可用性も求められています。最も適切な構成はどれですか？

A. ヨーロッパリージョンをプライマリに、アメリカリージョンをスタンバイにしてData Guardを構成する  
B. ヨーロッパリージョン内でAutonomous Data Guardを構成し、複数ADにスタンバイを配置する  
C. GoldenGateを使ってヨーロッパとアジアリージョン間でリアルタイム複製を行う  
D. OCI Full Stack Disaster Recoveryを使ってグローバルに分散する

<details>
<summary>▶ 解答・解説</summary>

**正解: B**

データ主権法が適用される場合、データを国外（別リージョン）に出すことが法的に禁止される。そのため**同一リージョン内**での高可用性構成（Autonomous Data Guard＋マルチAD）が唯一の選択肢。

- A、C、D: いずれもデータが別のリージョン（別の法域）に移動するため、データ主権法に違反するリスクがある

**重要ポイント**: **「データ主権」が出たら、クロスリージョン構成を選ばない**
</details>

### Q5
OCI Object Storageでユーザー管理の暗号化キー（OCI Vault）を使用してデータを暗号化し、クロスリージョン・レプリケーション（CRR）を構成しました。しかしリージョン障害後、複製先リージョンのデータにアクセスできませんでした。最も可能性の高い原因はどれですか？

A. CRRの設定が不完全でデータが複製されていなかった  
B. IAMポリシーが障害時に自動変更された  
C. ソースリージョンの暗号化キーがターゲットリージョンで利用できず、データを復号できない  
D. ライフサイクルポリシーがデータを削除した

<details>
<summary>▶ 解答・解説</summary>

**正解: C**

OCI Vaultの暗号化キーは**リージョン固有のリソース**。ソースリージョンで暗号化されたデータは、ターゲットリージョンに対応するキーと復号権限がないと読み取れない。データは物理的にコピーされていても、キーがなければ「アクセス不能」となる。

DR設計では、暗号化キーのレプリケーション（または各リージョンへのインポート）を必ず含める必要がある。

**重要ポイント**: **CRR ＋ カスタムKMS = ターゲットリージョンへのキー管理が必須**
</details>

### Q6
以下のDR戦略の説明のうち、正しくないものはどれですか？

A. Oracle RACはリージョン間に分散配置してDRを構成できる  
B. Data Guardのスタンバイは別リージョンに配置してリージョン障害に備えられる  
C. Object StorageのCRRはデータ耐久性と広域DRの両方に活用できる  
D. GoldenGateはアクティブ・アクティブのマルチリージョン同期に使用できる

<details>
<summary>▶ 解答・解説</summary>

**正解（誤りの記述）: A**

Oracle RACは複数ノードが**単一の共有ストレージ**にアクセスするアーキテクチャであり、ノード間インターコネクトに極めて低いレイテンシが必要。地理的に離れたリージョン間での構成は技術的に不可能またはサポート外。

リージョン間DRには**Data Guard**または**GoldenGate**を使用する。

**重要ポイント**: **RAC = リージョン内の可用性。リージョン間DR = Data Guard / GoldenGate**
</details>

### Q7
企業がRPO 4時間・2TBのデータベースに対してコスト効率の高いDR戦略を求めています。最も適切な選択肢はどれですか？

A. Active Data Guardをクロスリージョンで構成し、リアルタイム同期する  
B. GoldenGateでリアルタイムレプリケーションを行い、Object Storageにログを保存する  
C. 4時間ごとに変更分をスクリプトでエクスポートしてObject Storageに保存し、Functionsで別リージョンに転送する  
D. 同一リージョン内の別ADにBlock Volumeバックアップを取得する

<details>
<summary>▶ 解答・解説</summary>

**正解: C**

RPO 4時間という「緩やかな」要件に対して、Data GuardやGoldenGateは「ほぼリアルタイム同期」を実現するが、スタンバイDBの常時稼働コストが発生する（オーバーキル）。4時間ごとの差分転送 + Object Storage保管 + Functionsによる自動化は、要件を満たしつつコストを最小化できる。

- A、B: 技術的には優れているが、RPO 4時間に対してコストが過剰
- D: 同一リージョン内では、リージョン全体の災害からデータを保護できない

**重要ポイント**: **DR設計 = RPO/RTO要件 × コストのバランス。緩い要件に高コスト技術は不適切**
</details>

### Q8
2ノードOracle RAC構成のプライマリと、別リージョンの2ノードRACスタンバイをOracle Active Data Guardで接続した構成があります。この構成の説明として**正しいもの**を2つ選んでください。

A. プライマリリージョンのADが1つ落ちても、RAC構成により業務を継続できる  
B. プライマリリージョン全体が落ちた場合、スタンバイRACが自動的に引き継ぐ  
C. スタンバイRACから読み取り専用のクエリを実行できる  
D. RACはリージョン間のネットワークレイテンシに関係なく動作する  
E. GoldenGateを追加しないとクロスリージョンのデータ同期はできない

<details>
<summary>▶ 解答・解説</summary>

**正解: A、C**

- A: 各リージョン内のRAC構成により、ノード障害・AD障害に対してリージョン内での業務継続が可能
- C: **Active** Data Guardの特徴は、スタンバイからの読み取り専用アクセスが可能なこと

- B: 自動フェイルオーバーはActive Data Guardの設定によるが、一般的にリージョン間フェイルオーバーは自動とは限らない（手動介入が必要なケースもある）。設問の文脈では厳密に「自動」と断言できない
- D: RACはノード間の低レイテンシが前提。リージョン間の高レイテンシでは動作しない
- E: Data Guard自体がクロスリージョンのデータ同期を担う。GoldenGateは別途不要

**重要ポイント**: **RAC = リージョン内HA、Active Data Guard = スタンバイ読取 + リージョン間DR**
</details>

### Q9
急速に成長しているECサイトが、トランザクション処理と一部のリアルタイム分析を1つのDBでまかないたいと考えています。運用の自動化も重視しています。最適なOCIサービスはどれですか？

A. OCI Autonomous Data Warehouse（ADW）  
B. OCI Autonomous Transaction Processing（ATP）  
C. OCI MySQL Database Service  
D. Oracle Base Database（VM）

<details>
<summary>▶ 解答・解説</summary>

**正解: B**

ATPは「OLTP ＋ 混合ワークロード」に対応し、自動チューニング・自動インデックス・自動スケーリングを備える。リアルタイム分析も同一DB上で実行可能。

- A: ADWは分析特化。頻繁なOLTP更新には不向き
- C: MySQL DBSは自律型（Autonomous）の自動管理機能レベルはATPより低い
- D: Base DBは手動管理が必要で「運用の自動化」の要件に反する

**重要ポイント**: **「運用自動化 ＋ 混合ワークロード」= ATP**
</details>

### Q10
Autonomous Database（ADB）に関する説明として**正しくないもの**を2つ選んでください。

A. ADBは標準機能として自動バックアップを提供しており、手動でのバックアップは不要  
B. ADBへの接続には固定IPアドレスを使用するのがベストプラクティスである  
C. Autonomous Data Guardを使用することで、リージョン間の自動フェイルオーバーが可能  
D. ADBは大量の画像や動画を直接保存するのに最適なサービスである  
E. ADBのオートスケーリングは、OCPUを需要に応じて自動的に増減する

<details>
<summary>▶ 解答・解説</summary>

**正解（誤りの記述）: B、D**

- B: ADBへの接続は**ウォレット（Wallet）＋接続文字列（TNS）**を使用するのがベストプラクティス。固定IPへの依存はフェイルオーバー時に接続失敗を招く
- D: ADBはリレーショナルDBであり、画像・動画などの**非構造化データの大量保存**には不適切。これらはObject Storageに保存し、メタデータのみをADBで管理するのが正しい設計

- A: 正しい。ADBは60日間の自動バックアップを標準提供
- C: 正しい。Autonomous Data Guardでリージョン間DRが可能
- E: 正しい。ATPのオートスケーリングはOCPUを自動調整する

**重要ポイント**: **ADB接続 = ウォレット。画像・動画 = Object Storage（ADB不可）**
</details>

### Q11
金融機関がOracle Database 19cを3ノードRACで運用しています。高可用性・スケーラビリティ・ゼロデータロスを維持しながらOCIに移行するための最適な戦略はどれですか？

A. OCI VMインスタンスにDBをリフト＆シフトし、Data Guardをセットアップする  
B. Autonomous Transaction Processing（ATP）に移行する  
C. OCI Exadata DB Systemを使用し、RAC構成を含める  
D. OCI Base Database Serviceのシングルノードに移行し、RMANでバックアップする

<details>
<summary>▶ 解答・解説</summary>

**正解: C**

オンプレミスのRAC環境をそのまま（またはより高性能に）移行できるのは**Exadata DB System**だけ。RACサポートにより既存の高可用性構成を維持でき、Data Guardで追加のDR保護も可能。

- A: VMインスタンス上でRACを構築することは可能だが、Exadataほどのパフォーマンス最適化や管理機能がない
- B: ATPは有力だが、既存RACのカスタマイズや複雑な設定を完全に再現できない場合がある。「シームレス」な移行先としてはExadataが適切
- D: シングルノードではRACの高可用性要件を満たせない

**重要ポイント**: **オンプレRAC移行 = Exadata DB System**
</details>

### Q12
グローバル小売企業がJavaアプリケーションとMySQLデータベースをOCIに移行します。高可用性・スケーラビリティ・災害復旧が必要です。最も適切な構成はどれですか？

A. アプリをOCI Functionsに、MySQLをData Guard付きMySQL Database Serviceに移行する  
B. アプリをOCI Compute+オートスケーリングに、MySQLをMySQL Database Service+クロスリージョン・レプリケーションに移行する  
C. アプリをOKEに、MySQLをMySQL Database Serviceに移行し、IAMでセキュリティを設定する  
D. アプリをOCI VMに手動スケーリングで、MySQLをExadataで運用する

<details>
<summary>▶ 解答・解説</summary>

**正解: B**

- スケーラビリティ: Computeのオートスケーリングでピーク需要に自動対応
- DR: MySQL Database ServiceのクロスリージョンバックアップまたはReadレプリカでDR対応

重要な落とし穴: **Data GuardはOracle Database専用技術**。MySQL Database Serviceには使用できない（A が誤りの理由）。

- A: 技術的に不可能（MySQL ≠ Data Guard対象）
- C: DRへの言及がなく要件を満たさない
- D: 手動スケーリングはピーク対応不可

**重要ポイント**: **MySQL DR = Data Guardではなく、レプリカ/クロスリージョンバックアップ**
</details>

### Q13
OCI上で複数のAutonomous DatabaseとVM上のOracle Databaseが混在する環境でパフォーマンスの問題が発生しています。最小ダウンタイムで効果的に対処するための2つのアクションはどれですか？

A. すべてのDBを単一のAutonomous Databaseに統合する  
B. OCI Database Management Serviceを使用してフリート全体を統合監視する  
C. 問題のあるDBのCPUとストレージを分析なしで増強する  
D. VM上のOracle Databaseで自動メモリ管理（AMM）を有効にする  
E. Autonomous DBにデータベースシャーディングを実装する

<details>
<summary>▶ 解答・解説</summary>

**正解: B、D**

- B: **Database Management Service**はAutonomousとVM DBを問わず統合監視・管理できる唯一のサービス。Performance Hubを活用して問題箇所を特定できる
- D: VM上のDBでAMMを有効にすることで、SGA/PGAのメモリ配分がワークロードに合わせて動的最適化される

- A: 統合はリソース隔離要件に反し、リスクが高い
- C: 分析なしのリソース増強はコスト増だけで根本解決にならない（SQLの問題なら意味がない）
- E: シャーディングは超大規模環境用。ADBの自動スケーリングで十分な場合が多い

**重要ポイント**: **異種混在DB管理 = Database Management Service**
</details>

### Q14
大規模な金融アプリケーションのDBとして、インスタンス間の**強一貫性**と**高可用性**の両方が求められています。最も適切なOCIデータベース構成はどれですか？

A. Autonomous Transaction Processing（ATP）マルチリージョン展開  
B. Oracle RAC（複数の可用性ドメインにわたる構成）  
C. Oracle Data Guard（Maximum Performance モード）  
D. MySQL Database Service（HeatWave有効）

<details>
<summary>▶ 解答・解説</summary>

**正解: B**

Oracle RACは複数のインスタンスが**単一の共有ストレージ**にアクセスするため、インスタンス間でデータ不一致が生じない（**強一貫性**）。複数ADに配置することでAD障害にも耐えられる高可用性を実現。

- A: マルチリージョン展開はDR目的が主。ネットワーク遅延の関係でインスタンス間の強一貫性には不向き
- C: Data Guardはプライマリとスタンバイの構成でありRACとは異なる。インスタンス間の書き込み一貫性の問題には対応しない
- D: HeatWaveは分析高速化ツール。金融グレードの強一貫性と高可用性の実績ではRACに及ばない

**重要ポイント**: **インスタンス間の強一貫性 = Oracle RAC（共有ストレージ構成）**
</details>

### Q15
データウェアハウス用途で、大量データの分析クエリ・リアルタイムレポート・機械学習機能を1つのサービスでまかなう必要があります。最適なOCIサービスはどれですか？

A. OCI Big Data Service（Hadoop/Spark環境）  
B. OCI Data Science  
C. OCI Autonomous Data Warehouse（ADW）  
D. OCI Object Storage + OCI Data Flow

<details>
<summary>▶ 解答・解説</summary>

**正解: C**

ADWは分析特化の自律型DBで、インメモリ処理による高速分析クエリに加え、**Oracle Machine Learning（OML）をDBに内蔵**。SQLを使ってデータ移動なしにMLモデルの構築・予測が可能。分析・リアルタイム・MLの3要素を1サービスで満たせる。

- A: Big Data Serviceはバッチ処理向け。SQLの即時分析や組み込みMLでADWに劣る
- B: Data Scienceはモデル構築専用でDB機能・リアルタイムクエリを持たない
- D: Object StorageはストレージのみでSQL分析機能がない。Data Flowと組み合わせても即時性でADWに劣る

**重要ポイント**: **分析 ＋ ML ＋ 自動運用 = ADW（OML内蔵）**
</details>

### Q16
オンプレミスのOracle Database（20TB）をOCIに移行します。ダウンタイムを最小化し、移行中のデータ整合性を確保する最適な方法はどれですか？

A. RMAN（Recovery Manager）でバックアップ後にOCI DBへリストアする  
B. Oracle Data Pumpでエクスポートし、OCI DBへインポートする  
C. OCI Database Migration Service（DMS）の最小ダウンタイムオプションを使用する  
D. SCPでデータベースファイルをOCI Block Volumeに手動コピーする

<details>
<summary>▶ 解答・解説</summary>

**正解: C**

### Q37

ATP-Sデータベースでビジネス時間帯のCPUピーク需要に対応するための最も適切な方法はどれですか？

B. ピーク時に必要な最大CPU容量を事前に特定し、常にその数のCPUコアを設定しておく
