# モデルケースと評価方法

## 1. AI難易度レベル定義

### 難易度の考え方

AIにとっての難易度は「**言語→SQLまたは検索クエリへの変換の複雑さ**」で決まる。
3段階で分類し、**レベル1が合格してからレベル2へ、レベル2が合格してからレベル3へ**進む。

### Cortex Analyst の難易度軸

| レベル | 定義 | 典型的なSQL難易度 | AIの難しさの原因 |
|---|---|---|---|
| **L1 基礎** | 単一テーブル × 単一フィルタ条件 | `SELECT ... WHERE 列 = '値'` | ほぼなし。カラム名の認識のみ |
| **L2 応用** | 複数フィルタ条件（AND/OR）・集計・業務用語の解釈 | `WHERE A=1 AND B=1` / `GROUP BY` | 「使用している」→ 複数フラグのOR、「かつ」→ AND の正確な解釈 |
| **L3 発展** | テーブルJOIN・あいまいな表現・複合集計 | `JOIN ... ON` / 多段条件 | 2テーブルをまたぐ結合キーの推定、ビジネス表現の多義性 |

### Cortex Search の難易度軸

| レベル | 定義 | 検索の難しさ | AIの難しさの原因 |
|---|---|---|---|
| **L1 基礎** | ファイル名・モジュール名を明示した直接検索 | ai_summaryへの直接マッチ | ほぼなし。識別子が明示されている |
| **L2 応用** | 目的・処理内容から対象を推測する意味的検索 | セマンティックマッチ必要 | 「処理フロー」「条件」など概念的な問いへの対応 |
| **L3 発展** | 多段推論・障害原因の推定・複数コードの依存関係分析 | 複数文書の横断・推論 | 「なぜ〇〇になるか」という因果関係の推論 |

## 2. 難易度別 検証フロー

```
┌──────────────────────────────────────────────────────────────┐
│  L1（基礎）検証                                              │
│  対象: 単純影響調査 + ファイル名直接検索 + 基本解説          │
│  合格基準:                                                   │
│    ・影響調査 正解率 ≥ 85%                                   │
│    ・コード解説 平均スコア ≥ 4.0 / 5                        │
└──────────────────┬───────────────────────────────────────────┘
                   │ 合格後 → L2へ進む
                   │（L1精度は以降も問題ない前提）
                   ▼
┌──────────────────────────────────────────────────────────────┐
│  L2（応用）検証                                              │
│  対象: 複数条件・集計・意味的解説・基本的な複合質問          │
│  合格基準:                                                   │
│    ・影響調査 正解率 ≥ 80%                                   │
│    ・コード解説 平均スコア ≥ 3.5 / 5                        │
│    ・ツール選択精度（複合質問） ≥ 90%                        │
└──────────────────┬───────────────────────────────────────────┘
                   │ 合格後 → L3へ進む
                   │（L1・L2精度は以降も問題ない前提）
                   ▼
┌──────────────────────────────────────────────────────────────┐
│  L3（発展）検証                                              │
│  対象: テーブルJOIN・障害推論・複雑な複合質問               │
│  合格基準:                                                   │
│    ・影響調査 正解率 ≥ 70%                                   │
│    ・コード解説 平均スコア ≥ 3.0 / 5                        │
│    ・ツール選択精度（複合質問） ≥ 85%                        │
└──────────────────────────────────────────────────────────────┘
```

> **不合格時の対応**
> - L1不合格 → セマンティックモデルのsynonyms・verified_queriesを見直す
> - L2不合格 → システムプロンプトのfew-shot例追加、セマンティックモデルの条件定義を強化
> - L3不合格 → relationships（JOIN CLUE）定義の見直し、または「L3は人手+AI補助」の運用に切り替え検討

## 4. カテゴリA: 影響調査（単純）— 10問

### 評価方法
- **正解判定**: 期待するジョブ名/プログラム名のセットと一致するか
- **再現率**: 正解セットのうち何件を回答に含めたか
- **適合率**: 回答のうち実際に正解だった割合

### AI難易度の判定基準（Analyst）
- `L1`: 単一カラムの単一等価フィルタ（WHERE 列 = '値'）
- `L2`: ビジネス用語→複数フラグのOR/AND解釈、または集計が必要

| # | 難易度 | 質問 | 難易度の理由 | 期待する回答の型 |
|---|---|---|---|---|
| A-01 | L1 | 「〇〇テーブルを使用しているジョブをすべて教えて」 | 「使用している」= CRUD任意フラグのOR。単純だがOR解釈の確認が必要 | ジョブ名リスト |
| A-02 | L1 | 「〇〇テーブルを更新（UPDATE）しているプログラムは？」 | crud_u_flag=1 の単一条件 | src_nameリスト |
| A-03 | L1 | 「〇〇テーブルに対してINSERTしているジョブを教えて」 | crud_c_flag=1 の単一条件 | ジョブ名リスト |
| A-04 | L1 | 「〇〇テーブルをDELETEしているジョブは？」 | crud_d_flag=1 の単一条件 | ジョブ名リスト |
| A-05 | L1 | 「〇〇ジョブが参照しているテーブル一覧を教えて」 | ajs_name=〇〇 の単一条件 | テーブル名リスト |
| A-06 | L1 | 「〇〇ジョブネットに含まれるジョブをすべて教えて」 | net_name=〇〇 の単一条件 | ジョブ名リスト |
| A-07 | L2 | 「〇〇モジュールに含まれる機能の一覧を教えて」 | DISTINCT function_name + GROUP BY が必要 | 機能名リスト |
| A-08 | L1 | 「ファイル出力が発生する機能はどれですか？」 | file_o_flg=1 の単一条件 | 機能名リスト |
| A-09 | L1 | 「CSVファイルを取り込んでいるジョブを教えて」 | file_i_flg=1 の単一条件 | ジョブ名リスト |
| A-10 | L1 | 「〇〇.sqlを実行しているジョブ名を教えて」 | src_name=〇〇 の単一条件 | ジョブ名リスト |

## 6. カテゴリC: ソースコード解説 — 10問

### 評価方法（5点満点の人手評価）

| スコア | 基準 |
|---|---|
| 5 | 質問の意図に完全に答えており、ソースコードの関連箇所も明示されている |
| 4 | 質問の意図を概ね満たしており、軽微な不足がある |
| 3 | 部分的に回答しているが、重要な情報が欠けている |
| 2 | 関連する情報を含むが、質問への直接的な回答になっていない |
| 1 | 質問と無関係な回答 |

### AI難易度の判定基準（Search）
- `L1`: ファイル名・モジュール名が明示 → ai_summaryへの直接マッチ
- `L2`: 目的・フロー・注意点など概念的な問い → 意味的解釈が必要

| # | 難易度 | 質問 | 難易度の理由 | 評価ポイント |
|---|---|---|---|---|
| C-01 | L1 | 「〇〇.sqlはどんな処理をしているプログラムですか？」 | ファイル名明示 → ai_summaryの直接引用で回答可 | 処理概要の正確な説明 |
| C-02 | L2 | 「〇〇モジュールの処理内容を教えてください」 | モジュール全体を横断した複数件の情報統合が必要 | モジュール全体の説明 |
| C-03 | L2 | 「〇〇機能の処理フローを順を追って説明してください」 | 処理の順序性・因果関係の把握が必要 | 処理順序の正確性 |
| C-04 | L1 | 「〇〇.sqlが参照しているテーブルはどれですか？（コードから）」 | ファイル名明示 → source_codeから抽出するだけ | コードベースの特定 |
| C-05 | L2 | 「〇〇バッチはどんなデータを処理していますか？」 | データの種類・フローをコードから推測する必要 | データフローの説明 |
| C-06 | L1 | 「〇〇プログラムの処理を新人向けに分かりやすく説明してください」 | ファイル名明示 + 説明の平易化（LLMの基本能力で対応可） | 説明の分かりやすさ |
| C-07 | L2 | 「〇〇.sqlの中で最も重要な処理は何ですか？」 | ファイル名明示だが「重要度の判断」という主観的解釈が必要 | 処理の重要度判断 |
| C-08 | L2 | 「〇〇機能はどのような条件で動作しますか？」 | 条件分岐・トリガー条件をコードから読み取る必要 | 条件・トリガーの説明 |
| C-09 | L2 | 「〇〇モジュールの処理で注意すべき点はありますか？」 | リスク・注意点の抽出にはコード理解 + 判断が必要 | リスク・注意点の抽出 |
| C-10 | L1 | 「〇〇プログラムの概要をまとめてください」 | ファイル名明示 → ai_summaryそのままで回答可 | 概要の網羅性 |

## 9. 対応困難クエリ（現行設計の限界ケース）

> 以下のクエリは現在の2ツール構成では**完全な一発回答が困難**。
> 対応策は `03_orchestration_design.md` Section 7 を参照。

| # | 質問 | 困難な理由 | 現状での最善対応 |
|---|---|---|---|
| X-01 | 「〇〇.datファイルの6カラム目の桁数変更で改修・テストが必要な機能を教えて」 | ファイルのカラム定義情報がメタデータテーブルに存在しない。ai_summaryもカラム位置を記録していない可能性が高い | マルチターンで段階的に質問する（4ターン）か、ai_summaryの生成プロンプト改善で対応 |
| X-02 | 「〇〇APIの仕様変更（戻り値の型変更）で影響するプログラムは？」 | API呼び出し箇所・戻り値の型情報がメタデータに存在しない | source_code LIKE '%API名%' で呼び出し箇所は特定できるが戻り値の型チェック箇所まで追えない |
| X-03 | 「〇〇テーブルのカラム定義変更（NOT NULL化）で影響するプログラムは？」 | NULLチェックの実装の有無はソースコード読み取り + 推論が必要 | Searchで「NULLチェック」「〇〇テーブル」を組み合わせた検索で部分対応 |

**共通の根本課題**: 仕様変更影響の特定には「どのプログラムがその項目を**どのように**使っているか」という**実装レベルの情報**が必要。現在のメタデータ（CRUD/SRC）はファイル・テーブルの存在は管理しているが、カラムレベル・引数レベルの粒度は管理していない。

## 10. 評価スコアリング方法（自動・人手）

### 影響調査（カテゴリA・B）スコアリング

```
正解率 = 正解ケース数 / 全ケース数 × 100 (%)

再現率 = 正解セットのうち回答に含まれた件数の平均
       = Σ(正解件数に含まれた件数 / 正解全件数) / ケース数 × 100 (%)

適合率 = 回答のうち正解だった件数の平均
       = Σ(正解件数に含まれた件数 / 回答件数) / ケース数 × 100 (%)
```

### ソースコード解説（カテゴリC・D）スコアリング

```
平均関連性スコア = Σ(個別スコア) / ケース数  (1〜5点)
```

### ツール選択精度（カテゴリE）スコアリング

```
ツール選択精度 = 期待どおりのツールを使用したケース数 / 全ケース数 × 100 (%)
```

## 13. 評価実施手順

### Step 1: テスト環境準備

```sql
-- テストケース記録テーブルを作成（難易度レベルカラム追加）
CREATE OR REPLACE TABLE T_EVAL_RESULTS (
    eval_id          NUMBER AUTOINCREMENT,
    test_case_id     VARCHAR(10),    -- 例: A-01
    category         VARCHAR(5),
    difficulty_level INTEGER,        -- 1=L1基礎 / 2=L2応用 / 3=L3発展
    question         VARCHAR(2000),
    tool_type        VARCHAR(50),    -- search_only / hybrid
    used_tool        VARCHAR(200),   -- 実際に使われたツール
    answer           VARCHAR(20000),
    expected         VARCHAR(20000),
    is_correct       BOOLEAN,
    recall_score     FLOAT,
    precision_score  FLOAT,
    relevance_score  INTEGER,        -- 1〜5 (C/D/E)
    tool_match       BOOLEAN,
    response_sec     FLOAT,
    eval_date        TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
);
```

### Step 2: 難易度別に自動実行する

実行可能なスクリプトは `scripts/evaluate_agent.py` を参照。

```bash
# 環境変数を設定
export SF_ACCOUNT=xxx.ap-northeast-1.aws
export SF_USER=your_user
export SF_PASSWORD=your_password
export SF_DATABASE=YOUR_DB
export SF_SCHEMA=YOUR_SCHEMA
export SF_WAREHOUSE=YOUR_WH

# L1から順番に実行（hybrid）
python scripts/evaluate_agent.py --tool_type hybrid

# 比較用: Search only で同じテストケースを実行
python scripts/evaluate_agent.py --tool_type search_only

# L2から再開したい場合
python scripts/evaluate_agent.py --tool_type hybrid --level 2
```

**スクリプトの動作**:
1. `TEST_CASES` リストから指定レベルのケースを取得し、`SP_CHATBOT_AGENT` を順次呼び出す
2. 各ケースの回答・使用ツール・応答時間を `T_EVAL_RESULTS` にINSERT
3. レベル終了後に集計して合格基準と比較し、不合格なら停止・次レベルに進まない
4. C/D/Eカテゴリの `relevance_score` は人手評価後にUPDATEして再集計する（後述）

### Step 3: 人手評価（C・D・Eカテゴリ）

- 評価者2名が `T_EVAL_RESULTS` の `answer` カラムを確認し、5点満点でスコアをつける
- スコア差が2点以上の場合は協議して確定
- スコアが決まったら以下のSQLで更新する

```sql
-- 人手評価スコアをUPDATEする
UPDATE T_EVAL_RESULTS
SET relevance_score = <スコア値>  -- 1〜5
WHERE eval_id = <eval_id>;        -- T_EVAL_RESULTSのevalIdを指定

-- まとめてUPDATEする場合（Excelで集計してValuesで流し込む例）
UPDATE T_EVAL_RESULTS SET relevance_score = 4 WHERE eval_id = 10;
UPDATE T_EVAL_RESULTS SET relevance_score = 5 WHERE eval_id = 11;
-- ...
```

- UPDATE後にスクリプトの `--level` オプションで当該レベルを再集計して合格判定できる
- 影響調査（A・B）は `expected` カラムに正解リストを設定すれば自動判定される

### Step 4: 難易度別集計

```sql
-- 難易度別スコアを集計
SELECT
    difficulty_level,
    tool_type,
    COUNT(*) AS total_cases,
    ROUND(AVG(CASE WHEN is_correct THEN 1.0 ELSE 0.0 END) * 100, 1) AS accuracy_pct,
    ROUND(AVG(recall_score) * 100, 1) AS avg_recall_pct,
    ROUND(AVG(relevance_score), 2) AS avg_relevance_score,
    ROUND(AVG(response_sec), 2) AS avg_response_sec
FROM T_EVAL_RESULTS
GROUP BY difficulty_level, tool_type
ORDER BY difficulty_level, tool_type;
```
