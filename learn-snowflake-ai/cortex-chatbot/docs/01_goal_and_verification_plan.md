# ゴール設定と検証ステップ

## 1. 背景と課題

### 現状

- システム資産（ソースコード・バッチ・ジョブネット）のメタデータをSnowflakeテーブルに集約済み
- Cortex Searchとエージェントを用いて「〇〇テーブルを使用しているジョブを教えて」等の検索を試験中
- **課題**: テキスト類似検索のみでは、テーブル名の表記揺れや検索漏れが発生しやすい

### 仮説

| 質問タイプ | 現状 (Search only) | 提案 (Analyst + Search) |
|---|---|---|
| 影響調査（構造化クエリ） | テキスト類似検索 → 漏れ・過検出あり | SQL生成 → 完全一致保証 |
| ソースコード解説（非構造化） | ai_summaryへのRAG検索 → そのまま活用 | Cortex Search継続 + 結果精度向上 |
| 複合質問 | 1ツールで対応 → 精度低下 | ルーティングで最適ツール選択 |

---

## 2. ゴール設定

### 最終ゴール

> **Cortex Analyst + Cortex Search のハイブリッドエージェントが、単独Cortex Searchに比べて影響調査精度を向上させることを定量的に示す**

### 定量的評価指標 (KPI)

| 指標 | 現状ベースライン | 目標値 | 計測方法 |
|---|---|---|---|
| **影響調査 正解率** | Search only で計測 | +20% 以上向上 | モデルケース正解数 / 全ケース数 |
| **影響調査 再現率** | Search only で計測 | +15% 以上向上 | 正解に含まれた件数 / 正解全件数 |
| **ソースコード解説 関連性スコア** | Search only で計測 | 同等以上 (±5% 以内) | 5点満点の人手評価平均 |
| **エージェント応答時間** | 計測なし | 10秒以内（P95） | API応答タイムスタンプ差分 |
| **ツール選択精度** | 該当なし | 90% 以上 | 意図どおりのツール使用率 |

### 成功基準

- [ ] 影響調査の正解率が Search only 比 **+20% 以上** 向上している
- [ ] ソースコード解説の関連性スコアが Search only と **同等以上** である
- [ ] エージェントが影響調査クエリと解説クエリを **90% 以上の精度** で正しくルーティングしている
- [ ] Streamlitアプリで実際の質問に回答でき、SQL結果が表示される

## 3. 検証ステップ

### フェーズ 1: ベースライン計測（Search only）

**目的**: 現状のCortex Search単独の精度を定量化する

| ステップ | 作業内容 | 担当 | 成果物 |
|---|---|---|---|
| 1-1 | モデルケース30問を定義する（影響調査15問 / 解説15問） | 人手 | テストケース一覧 |
| 1-2 | 各テストケースの期待回答（正解）を定義する | 人手 | 正解セット |
| 1-3 | 現行Search onlyエージェントで全30問を実行する | 自動 | 回答ログ |
| 1-4 | 正解率・関連性スコア・再現率を集計する | 人手+自動 | ベースラインスコア |

**期間目安**: 1〜2日

---

### フェーズ 2: Cortex Analystセットアップ

**目的**: T_システム名_CRUDテーブルのセマンティックモデルを構築・検証する

| ステップ | 作業内容 | 成果物 |
|---|---|---|
| 2-1 | セマンティックモデルYAMLを作成する（テーブル・カラム・ビジネス用語定義） | `02_semantic_model.yaml` |
| 2-2 | Cortex AnalystのAPIでモデルケース影響調査クエリ15問を実行する | 回答ログ |
| 2-3 | 生成SQLを人手確認し、モデル定義を調整する（verified queriesの登録） | 調整済みYAML |
| 2-4 | 影響調査の正解率・再現率を集計し、フェーズ1と比較する | 比較表 |

**期間目安**: 2〜3日

**検証観点**:
- 正しいテーブル名でSQLが生成されているか
- CRUDフラグのフィルタ条件が正しく解釈されているか（「更新している」→ `crud_u_flag = 1`）
- 「〇〇テーブルを使用している」= R/C/U/D いずれかが1 として解釈されるか

## 5. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| セマンティックモデルの解釈精度が低い | 影響調査の正解率が上がらない | verified queriesを積極活用してSQLを明示的に登録 |
| テーブル名の日本語表記揺れ | 検索漏れ | table_name_jpカラムの表記統一、synonyms定義 |
| Cortex Agentのツール選択ミス | ルーティング精度低下 | システムプロンプトでルーティング指針を明示 |
| source_codeカラムのサイズ過大 | Cortex Search インデックス効率低下 | ai_summaryを主検索対象、source_codeはフィルタ用途に限定 |
