# Snowflake Intelligence

## Snowflake Intelligence とは

Snowflake Intelligence は、**Cortex Agent の上に Snowflake がフロントエンド UI を被せたマネージドサービス**です。データソースを登録するだけで、コーディング不要のチャット型分析 UI が手に入ります。

> 「Cortex Agent = エンジン」  
> 「Snowflake Intelligence = エンジン ＋ 完成した車体」

---

## Cortex Search との統合について（重要）

「Intelligence は Cortex Search に対応していない」と思われがちですが、**正確には以下の通りです。**

| 接続方式 | Cortex Search 対応 | 説明 |
|---|---|---|
| **Semantic Model 直接接続** | ❌ 不可 | Cortex Analyst のみ動作。ドキュメント検索は対象外 |
| **Cortex Agent 経由** | ✅ 可能 | Agent のツールとして Cortex Search を組み込める |

```
【誤解】
Intelligence → Cortex Search（直接接続） … ❌

【正しいアーキテクチャ】
Intelligence → Cortex Agent → Cortex Analyst（構造化データ）
                            → Cortex Search  （ドキュメント検索）
```

**つまり、Agent をデータソースとして登録すれば、Intelligence から Analyst + Search の統合が実現できます。**

---

## 2つの利用パターン

| パターン | 構成 | 向いているケース |
|---|---|---|
| **シンプル構成** | セマンティックモデル直接接続 | 数値データの質問応答のみ必要な場合 |
| **統合構成（推奨）** | Cortex Agent 経由（Analyst + Search） | データ分析 + 社内ドキュメント調査を統合したい場合 |

---

## Streamlit in Snowflake との比較

| 比較軸 | Snowflake Intelligence | Streamlit in Snowflake |
|---|---|---|
| セットアップ | エージェント設定を登録するだけ | Python でアプリを実装する |
| コーディング | **不要** | 必要 |
| UI | Snowflake 提供のチャット UI（洗練済み） | 自前実装（自由度高い） |
| グラフ表示 | 自動（棒・折れ線・テーブル等） | 自前で Altair 等を実装 |
| Cortex Search 統合 | ✅（Agent 経由で対応） | ✅（実装次第） |
| カスタマイズ | 限定的（システムプロンプトで調整） | **完全に自由** |
| エンドユーザー共有 | **簡単**（URL 共有・Snowsight から直接） | Streamlit デプロイが必要 |
| 向いている用途 | **BIセルフサービス・エンドユーザー向け** | 開発者デモ・社内ツール構築 |

### Analyst → Search 連携パターンについて

#### 仕組み：ReAct ループによるツール間のコンテキスト共有

Cortex Agent は **ReAct（Reasoning + Acting）ループ**をサーバー側で実行しています。各ツールの実行結果はエージェントのコンテキストに蓄積され、次のツール呼び出しの入力として LLM が活用します。

```
ユーザー: 「先月の売上が最も低いカテゴリについて、社内施策を解説して」

┌─ Step 1: 計画 ──────────────────────────────────────────────┐
│  「まず Analyst でカテゴリを特定し、その値で Search を呼ぶ」 │
└──────────────────────────────────────────────────────────────┘
  ↓
┌─ Step 2: Analyst 実行 ───────────────────────────────────────┐
│  SQL: SELECT category, change_pct FROM ... ORDER BY ...      │
│  結果: { category: "家電", change_pct: -15.3 }               │
└──────────────────────────────────────────────────────────────┘
  ↓ 結果をコンテキストに保持
┌─ Step 3: 反省・次のステップ計画 ────────────────────────────┐
│  「category = "家電" を Search クエリに使う」                 │
└──────────────────────────────────────────────────────────────┘
  ↓
┌─ Step 4: Search 実行 ────────────────────────────────────────┐
│  Query: "家電カテゴリ 施策 キャンペーン"                      │
│  結果: "2024年Q4 家電カテゴリ施策" ドキュメント               │
└──────────────────────────────────────────────────────────────┘
  ↓
┌─ Step 5: 統合回答 ───────────────────────────────────────────┐
│  「家電は -15.3%。施策によると対象外製品の影響が主因…」       │
└──────────────────────────────────────────────────────────────┘
```

#### ポイント：列の値がどう Search クエリになるか

エージェントの LLM が Analyst の結果（行データ・列の値）を読み取り、**自然言語のクエリを再構成**して Search を呼びます。直接パラメータを渡すのではなく、LLM の理解を介して変換されます。

```
Analyst 結果（列の値）       Search クエリへの変換（LLM が行う）
─────────────────────────────────────────────────────────────────
category = "家電"       →  "家電カテゴリ 施策 キャンペーン 2024年"
product_name = "ノートPC Pro 15" → "ノートPC Pro 15 仕様 マニュアル"
region = "東日本"       →  "東日本 配送 物流 レポート"
```

#### 動作条件

| 条件 | 判断 | 補足 |
|---|---|---|
| 列の値がテキスト（カテゴリ名・商品名など） | ✅ 有効 | Search の意味検索が効きやすい |
| 列の値が数値 ID のみ（例: `product_id = 1042`） | ⚠️ 要工夫 | ID だけでは Search で意味検索できない。商品名などテキストの列も SELECT に含めること |
| Search Service に対応するコンテンツが存在しない | ❌ 結果なし | 「該当ドキュメントが見つかりません」という回答になる |
| 対象レコードが複数ある場合 | ⚠️ 要指定 | どのレコードの値を使うかをオーケストレーション手順か質問文で明示する |

#### 列の値を正しく Search に渡すためのセマンティックモデルの設定

Analyst の SQL 結果に**テキストの列**が含まれていることが重要です。セマンティックモデルで対象の列を `dimension` として定義しておいてください。

```yaml
# セマンティックモデルの例（抜粋）
dimensions:
  - name: category
    description: 商品カテゴリ名（家電・電子書籍・ゲームなど）
    expr: category
    data_type: TEXT      # ← テキスト型の列を Search クエリに使う

  - name: product_name
    description: 商品名
    expr: product_name
    data_type: TEXT
```

#### 連携パターンを引き出す質問例

```
✅ 効果的な質問パターン:
  「先月の売上が最も低いカテゴリについて、社内施策ドキュメントを解説して」
    → Analyst でカテゴリ特定 → そのカテゴリ名で Search

  「最も返品率が高い商品の仕様・マニュアルを調べて」
    → Analyst で商品名特定 → 商品名で Search

  「東日本地域の売上低下に関連する物流報告を見せて」
    → Analyst で東日本の数値確認 → "東日本 物流" で Search

⚠️ 機能しにくいパターン:
  「product_id=1042 について教えて」
    → ID だけでは Search が意味検索できない
    → 「最も売上の低い商品について…」のように自然言語で質問する
```

#### ① オーケストレーション手順（orchestration instructions）

**「どのツールをいつ使うか」の判断ルール**を書きます。エージェントがツールを選択する際の推論に使われます。

```
# ツール選択ルール

## sales_analyst（Cortex Analyst）を使う場面
- 売上・注文件数・売上高などの数値を知りたい場合
- 商品・カテゴリ・期間・地域などでデータを絞り込みたい場合
- 前月比・前年比など時系列比較が必要な場合
- ランキング・集計・トレンド分析が必要な場合

## doc_search（Cortex Search）を使う場面
- キャンペーン・施策の背景や詳細を調べる場合
- 配送・物流・在庫に関する社内報告や規定を確認する場合
- 売上数値の原因・背景を社内ドキュメントで裏付けたい場合

## 両方を組み合わせて使う場面（連携パターン）
- 「なぜ売上が下がったか」のように数値と背景の両方が必要な場合
- 施策の効果を定量的に検証したい場合
- 「〇〇について解説して」のように、まず対象レコードを特定してからドキュメントを検索する場合

## Analyst → Search の連携手順（重要）
構造化データで対象レコードを特定した後、そのレコードの特定カラムの値を
Search の検索クエリとして使用する場合は以下の手順で実行すること:
  1. sales_analyst で対象レコードを取得し、検索に使用するカラム値を確認する
  2. そのカラム値（カテゴリ名・商品名など）を含む自然言語クエリを構築する
  3. 構築したクエリで doc_search を呼び、関連ドキュメントを取得する
  4. Analyst の数値データと Search のドキュメントを統合して回答する

# 追加ルール
- ツール実行前に質問をどちらの分類に属するか判断してから実行すること
- 不明確な場合は両方のツールで確認を試みること
```

#### ② 応答手順（response instructions）

**「ツール実行後の回答をどう見せるか」**のフォーマット・トーンを書きます。ツール選択とは独立した設定です。

```
- グラフで表現できる数値データは、ユーザーが指定しなくてもグラフを生成すること
- 数値を引用する際は期間・集計軸を明示すること（例:「2024年11月のカテゴリ別では…」）
- 社内ドキュメントを引用する際はドキュメント名を明記すること
  （例:「東日本エリア配送遅延報告によると…」）
- データとドキュメントを組み合わせた場合は、それぞれの根拠を分けて示すこと
- 回答は簡潔に。箇条書きを積極的に使うこと
- 必ず日本語で回答すること
```

#### CREATE AGENT SQL（参考: コードで定義する場合）

UI 入力の代わりに SQL でエージェントを定義することもできます。

```sql
CREATE OR REPLACE AGENT ANALYST_DEMO_DB.ANALYST_SCHEMA.SALES_INTELLIGENCE_AGENT
  COMMENT = '売上データと社内ドキュメントを横断して分析するエージェント'
  FROM SPECIFICATION $$
    models:
      orchestration: claude-3-5-sonnet

    orchestration:
      budget:
        seconds: 30
        tokens: 16000

    instructions:
      orchestration: |
        # ツール選択ルール

        ## sales_analyst（Cortex Analyst）を使う場面
        - 売上・注文件数・売上高などの数値を知りたい場合
        - 商品・カテゴリ・期間・地域などでデータを絞り込みたい場合
        - 前月比・前年比など時系列比較が必要な場合

        ## doc_search（Cortex Search）を使う場面
        - キャンペーン・施策の背景や詳細を調べる場合
        - 配送・物流・在庫に関する社内報告や規定を確認する場合

        ## 両方を組み合わせる場面
        - 数値と原因の両方が必要な場合（例: 「なぜ売上が下がったか」）

      response: |
        - グラフで表現できる数値データは積極的にグラフ化すること
        - 数値を引用する際は期間・集計軸を明示すること
        - ドキュメントを引用する際はドキュメント名を明記すること
        - 必ず日本語で回答すること

    sample_questions:
      - question: "先月の売上が最も低下したカテゴリはどこですか？"
      - question: "家電カテゴリの売上低下の原因を調べてください"
      - question: "先月の売上上位5商品と、それに関連する社内施策を合わせて教えてください"
      - question: "東日本エリアの配送状況に関する社内報告を確認してください"

    tools:
      - tool_spec:
          type: cortex_analyst_text_to_sql
          name: sales_analyst
          description: "売上・注文・商品・顧客に関する構造化データを自然言語で分析する"
      - tool_spec:
          type: cortex_search_service
          name: doc_search
          # ↓ エージェントがいつこのツールを使うかを判断する重要な説明文
          description: >
            社内ドキュメント（キャンペーン施策・物流報告・社内規定）を全文検索する。
            売上数値の背景・原因調査、施策の詳細確認、配送・在庫に関する報告書の参照に使用する。

    tool_resources:
      sales_analyst:
        semantic_model_file: "@ANALYST_DEMO_DB.ANALYST_SCHEMA.SEMANTIC_MODEL_STAGE/03_semantic_model.yaml"
      doc_search:
        # Cortex Search Service の完全修飾名
        name: "ANALYST_DEMO_DB.ANALYST_SCHEMA.COMPANY_DOC_SEARCH"
        # 返す検索結果の最大件数
        max_results: "5"
        # 検索結果のタイトルとして表示するカラム（ATTRIBUTES に含まれるカラム）
        title_column: "doc_name"
        # ドキュメントを一意に識別するカラム
        id_column: "doc_id"
        # filter: カテゴリを絞り込む場合のみ設定（全カテゴリ対象なら省略）
        # filter:
        #   "@eq":
        #     category: "キャンペーン"
        # カラムの意味をエージェントに説明（フィルターを動的に使いこなすために推奨）
        columns_and_descriptions:
          - column: "doc_name"
            description: "ドキュメントのタイトル"
          - column: "category"
            description: "ドキュメントのカテゴリ。値は キャンペーン / 物流 / 社内規定 のいずれか"
          - column: "content"
            description: "ドキュメントの本文。キャンペーン施策・配送報告・社内規定の詳細が含まれる"
  $$;
```

#### Cortex Search ツール — パラメータ入力ガイド

Cortex Search をツールとして登録する際の全パラメータを整理します。

**tool_spec（ツールの識別と使い方の説明）**

| パラメータ | 役割 | デモでの入力値 | 入力指針 |
|---|---|---|---|
| `type` | ツール種別の固定値 | `cortex_search_service` | 変更不要 |
| `name` | エージェントがツールを参照する識別子 | `doc_search` | 英小文字・アンダースコアで命名 |
| `description` | **エージェントがいつこのツールを使うかを判断する根拠**（最重要） | 下記参照 | 「何を検索できるか」「どんな質問に使うか」を具体的に書く |

`description` の推奨記述:
```
社内ドキュメント（キャンペーン施策・物流報告・社内規定）を全文検索する。
売上数値の背景・原因調査、施策の詳細確認、配送・在庫に関する報告書の参照に使用する。
```

> ⚠️ `description` が曖昧だとエージェントがツールを呼ばないケースがあります。  
> 「何が入っているか」と「どんな質問に答えられるか」の両方を書くのがポイントです。

---

**tool_resources（ツールのリソース設定）**

| パラメータ | 役割 | デモでの入力値 | 入力指針 |
|---|---|---|---|
| `name` | Cortex Search Service の完全修飾名 | `ANALYST_DEMO_DB.ANALYST_SCHEMA.COMPANY_DOC_SEARCH` | `DB.SCHEMA.SERVICE_NAME` 形式で指定 |
| `max_results` | 返す検索結果の最大件数 | `5` | 3〜10 が目安。多いほど回答の根拠が増えるがトークン消費も増える |
| `title_column` | 検索結果のタイトルとして表示するカラム | `doc_name` | Search Service の ATTRIBUTES に含まれるカラムを指定 |
| `id_column` | ドキュメントを一意に識別するカラム | `doc_id` | PK カラムを指定。省略可だが設定推奨 |
| `filter` | 特定の属性値で結果を絞り込む | （省略 — 全カテゴリ対象） | 特定カテゴリのみにしたい場合に設定。動的フィルターを使いたければ `columns_and_descriptions` で十分 |
| `columns_and_descriptions` | 各カラムの意味をエージェントに説明 | 下記参照 | filterable/searchable なカラムに説明を付けると、エージェントが動的フィルターを使いこなせるようになる |

`columns_and_descriptions` の推奨記述（デモ用）:

| column | description |
|---|---|
| `doc_name` | ドキュメントのタイトル |
| `category` | ドキュメントのカテゴリ。値は `キャンペーン` / `物流` / `社内規定` のいずれか |
| `content` | ドキュメントの本文。キャンペーン施策・配送報告・社内規定の詳細が含まれる |

> `category` の説明に取りうる値を列挙しておくと、「キャンペーン施策だけ調べて」という質問でエージェントが自動的に `{"@eq": {"category": "キャンペーン"}}` フィルターを生成できます。

## デモシナリオ（5分でできる統合デモ）

前提: セマンティックモデル YAML + Cortex Search Service が作成済みの状態

```
1. Snowsight > AI & ML > Intelligence
2. + Create Intelligence App → 名前入力 → Create
3. + Add Data Source → Type: Agent を選択
4. ① オーケストレーション手順を入力（ツール選択ルールをコピー）
5. ② 応答手順を入力（グラフ優先・引用形式・言語をコピー）
6. ③ スキルは空欄のまま（設定不要）
7. ④ 質問の例を4件入力（スターター質問）
8. ツール追加: sales_analyst（Analyst）+ doc_search（Search）
9. ウェアハウスを設定 → Publish

所要時間: 約10〜15分（4フィールド入力込み）
```

**デモで見せたい質問の流れ:**

```
質問 1: 「月別の売上合計を教えて」
  → sales_analyst が動作 → 折れ線グラフでトレンド表示
  ※ Cortex Analyst だけが呼ばれることを確認

質問 2: 「東日本エリアの配送に関する社内報告を教えて」
  → doc_search が動作 → 配送遅延報告ドキュメントを引用
  ※ Cortex Search だけが呼ばれることを確認

質問 3: 「先月の家電カテゴリの売上低下の原因を調べて」
  → sales_analyst + doc_search の両方が動作
  ※ 2つのツールが組み合わさることをアピール

質問 4: 「2025年Q1の重点施策と、それに連動した売上計画は？」
  → doc_search で施策ドキュメント取得
  → sales_analyst で関連カテゴリのデータ取得
  ※ 複合タスクの自律解決を見せる
```

## できること・できないこと

### できること

- 自然言語 → SQL 生成 → 結果表示（Cortex Analyst の全機能）
- **Cortex Search によるドキュメント検索（Agent 経由）**← 追加
- グラフの自動選択・表示（棒・折れ線・テーブル）
- マルチターン会話
- SQL の透明性表示（`使用ツールを見る` の展開）
- システムプロンプトによるオーケストレーション制御
- Verified Queries の活用

### できないこと・制限

- UI デザインのカスタマイズ（ロゴ・色・レイアウト変更不可）
- Streamlit のような独自ロジック（承認フロー・通知など）の組み込み
- Cortex Search の **直接接続**（Agent 経由なら可）
- アプリの埋め込み（外部サイトへの iframe 埋め込みは非対応）

## 4サービスの使い分けまとめ（更新版）

| シナリオ | 推奨 |
|---|---|
| 今すぐデモしたい（数値データのみ） | **Snowflake Intelligence（シンプル構成）** |
| 数値データ + 社内ドキュメントを統合したい | **Snowflake Intelligence（Agent 構成）** |
| セマンティックモデルを GUI で作りたい | **Snowsight Semantic Models** |
| 自社ブランドのチャット UI が欲しい | Streamlit in Snowflake |
| 承認フローや通知などの独自ロジックが必要 | Streamlit in Snowflake |
| バックエンド API として Cortex Agent を使いたい | Cortex Agent REST API 直接呼び出し |
